# 安全設計の前提

このシステムは水道直結の 12V 常時閉ソレノイドバルブを扱います。したがって、一般的な Web アプリよりはるかに保守的に設計する必要があります。

# 水道直結のリスク

- バルブ固着や制御故障で連続通水すると、短時間で大量漏水になりうる
- 漏水は床、壁、電源、家具、近隣へ被害を広げる
- 土の飽和だけでなく、配管抜けやホース破断でも事故になる

このため「AI が間違えないこと」を前提にせず、「間違えても止まること」を前提にします。

# 常時閉ソレノイドバルブの意義

- 通電している間だけ開く
- 電源断時に閉じる
- Raspberry Pi 停止、Arduino 停止、クラウド停止の多くで閉じる側に倒れる

これはフェイルセーフの中心です。

# 手動バルブの必須化

- ソレノイドバルブの上流に、必ず手動バルブを付ける
- 実験外時間、外出時、就寝時は手動バルブを閉じる
- 緊急時はまず手動バルブを閉める

# 流量調整バルブの必須化

- 1 回の誤通水量を物理的に小さくするため、流量調整バルブを入れる
- ソフトウェアのミスだけで大量出水しないようにする
- 実測流量を `calibration_profiles` に反映する

# 水道接続前ゲート

次が完了するまで、水道接続した自動散水を禁止する。

- 手動バルブ設置
- 流量調整バルブ設置
- 流量測定
- wet 閾値の実土壌校正
- 12V 電源確認
- MOSFET / リレー発熱確認
- USB 抜去時の停止確認
- Raspberry Pi 停止時に再通水しない確認
- 電源断でソレノイドが閉じる確認
- `daily_limit_exceeded` の確認

このゲートが完了するまでは、Raspberry Pi からの `water` 自動実行を本番で有効化しない。

# ソフトウェア上限

- Cloud Run 側で提案量に上限をかける
- Raspberry Pi 側で mL -> ms 換算後に上限をかける
- Arduino 側で最終上限を強制する
- 1 回上限と 1 日累積上限の両方を設ける

同じ制限を 3 層で持つのは冗長ですが、安全上は意図した冗長です。

# Arduino側安全制御

- 起動時は必ず `closeValve()`
- `water` コマンド時は湿りすぎなら拒否する
- 設定上限を超える開放時間は切り詰める
- 1 日累積上限を超える場合は拒否する
- 不正コマンドは拒否する
- 応答不能や再起動でもバルブは閉じる設計にする
- 実バルブ未接続時は dry-run mode で TEST LED だけを点灯し、机上試験を先に行う
- 初期実装の日次上限は RTC 未導入のため「起動後 24 時間窓」で管理する
- 土壌水分の wet 判定は単発値ではなく複数回平均で行い、ノイズで誤って通水しにくくする

Arduino は単なる命令実行装置ではなく、水回りの安全装置として扱う。

# 実行モードの扱い

- `DRY_RUN_MODE=true`
  - Arduino は実バルブを開かず、机上試験専用とする
  - 水道接続前、配線確認、シリアル確認ではこの状態を維持する
- `DRY_RUN_MODE=false`
  - 実バルブを開ける可能性があるため、水道接続前ゲート完了後にだけ使う
  - 切替時は `ALLOW_WATER_COMMAND_FROM_PI` が意図どおりかを同時に確認する
- `ALLOW_WATER_COMMAND_FROM_PI=false`
  - Raspberry Pi からの `water` 自動送信を禁止する
  - 初期実装と観測先行運用ではこれを既定値とする
- `ALLOW_WATER_COMMAND_FROM_PI=true`
  - 流量測定、wet 閾値校正、安全試験、運用確認が終わるまで本番で有効化しない

# GCP障害時の挙動

- 観測は Raspberry Pi で継続する
- AI 判断ができないため、自動水やりは保留にする
- 既存の高頻度測定ジョブだけはローカルで継続する
- 障害復旧後にログを再送する

# Raspberry Pi停止時の挙動

- 新しい観測、判断要求、水やり命令は止まる
- Arduino に新規命令が届かないので通常は通水しない
- 実験者は手動バルブを閉めて確認する

# Arduino停止時の挙動

- 通常はバルブ閉になる
- Raspberry Pi は水やり失敗として記録する
- 復旧するまで自動水やりを無効にする

# 電源断時の挙動

- ソレノイドは閉じる
- Raspberry Pi の未送信ログはローカル保存から復旧させる
- 起動後は即水やりせず、まず状態確認を行う

# 通信断時の挙動

## Pi と GCP の通信断

- 観測は継続
- AI 判断は停止
- ローカルキューに保存

## Pi と Arduino の通信断

- 水やりは実行しない
- エラーを `watering_events` または `safety_events` に記録

# 漏水時の手順

1. 手動バルブを閉める
2. 12V 電源を切る
3. 周囲の電源タップを安全確認のうえ遮断する
4. Raspberry Pi サービスを停止する
5. `safety_events` に発生時刻、推定原因、被害範囲を記録する
6. 原因が特定できるまで再開しない

# 防水ボックス

- Raspberry Pi、電源、接続部は防水ボックスに入れる
- ただし密閉しすぎて熱がこもらないよう配慮する
- ケーブルの取り回しで水が伝わって侵入しないようにする

# 電源系統分離

- 12V 系はソレノイド専用に近い構成にする
- 5V 系は Raspberry Pi / Arduino 用に分ける
- ノイズで Pi が再起動してもバルブが開きっぱなしにならないようにする

# 12V系と5V系の分離

- リレーまたはドライバを介して制御する
- 逆起電力対策を入れる
- 共通 GND の取り方を確認し、誤配線を防ぐ

# 実験外時間の運用

- 実験外時間は手動バルブを閉める
- 長時間不在時はソレノイド系の 12V 電源も落とす
- 自動化より安全を優先し、夜間完全自動運転を前提にしない

# この段階で最低限必要な安全試験

- 起動時にバルブが閉じる
- `water` 実行中に USB を抜いても最大時間で止まる
- Pi を落としても再通水しない
- 12V 電源断で確実に閉じる
- wet 状態で `water` が拒否される
- `daily_limit_exceeded` が期待どおり拒否する
